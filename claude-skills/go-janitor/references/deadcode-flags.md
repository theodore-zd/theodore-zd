# deadcode flags cheat sheet

`deadcode` = `golang.org/x/tools/cmd/deadcode`. Install: `go install golang.org/x/tools/cmd/deadcode@latest`.

## Flags

| Flag | Use |
|------|-----|
| `-test` | Include test binaries (the hidden `main`s generated by `go test`). Default ON for this skill. |
| `-filter=regexp` | Restrict output to packages matching the regex. Useful in mono-repos. |
| `-whylive=pkg.Symbol` | Print the call chain proving a symbol IS reachable. Run when deadcode-says-live contradicts your suspicion. |
| `-json` | Machine-readable output for structured triage. |
| `-f=template` | Custom output template. Rare. |
| `-generated` | Include generated files. Leave OFF unless the user asks — generated code is owned by its generator. |

## Soundness

`deadcode` performs whole-program Rapid Type Analysis from entry points (`main`, `init`).

Sound against:
- Direct static calls
- Interface method dispatch (tracks types converted to each interface)
- `reflect.Value.Call()` over types reachable through interface conversions

NOT sound against:
- Callers in assembly (`.s` files)
- `//go:linkname` targets linked from other packages

If a flagged package uses CGo, assembly, or `//go:linkname`, require human review before removal.

## Common invocations

```bash
# Default sweep including tests
deadcode -test ./...

# Mono-repo: only one sub-module
deadcode -test -filter=example.com/foo/... ./...

# Confirm a symbol really is dead before deleting
deadcode -whylive=example.com/foo.Bar ./...

# Structured output for parsing
deadcode -test -json ./...
```
